# be-DailyJob

## 아키텍처 및 공통 환경 설정
### 폴더 구조 및 의존성 주입
코드 규모 확장과 다수 인원의 협업 시 발생할 수 있는 문제를 방지하고자, 계층형 구조가 아닌 응집도가 높은 도메인 중심 구조(Package by Feature)를 채택함. 의존성 주입은 테스트 코드 작성의 편의성을 고려해 생성자 주입(@RequiredArgsConstructor) 방식을 팀 내 표준으로 권장함.

### API 및 보안 설정
API 변경 시 기존 클라이언트에 미치는 영향을 최소화하기 위해 /api/v1 형태의 버저닝을 도입함. 인증/인가 로직은 서블릿 필터를 직접 구현하여 화이트리스트 기반 패턴 매칭을 적용할 계획임. API 문서화 툴은 구버전 라이브러리(Springfox) 대신, 현재 스프링 부트 환경에 적합한 springdoc-openapi 3.x 버전을 사용하기로 확정함.

## 응답 반환 구조 표준화
### ResponseBodyAdvice 채택
클라이언트의 처리 편의성과 통일성을 확보하기 위해 모든 API 응답 스키마(resultType, error, success)를 동일하게 규격화함. 개발자가 컨트롤러마다 직접 객체를 감싸는 방식은 누락 위험이 있으므로, 프레임워크 레벨에서 일괄 처리 가능한 ResponseBodyAdvice 방식을 최종 채택함. 이를 통해 프론트엔드에서는 Axios 인터셉터를 활용하여 전역 에러 처리를 효율적으로 수행할 수 있게 됨.

### 표준화 적용 시 주의 사항
예외 발생 시 ExceptionHandler와 Advice가 중복 적용되어 응답 데이터가 이중 포장되는 문제가 없도록 분기 처리가 필요함. 또한 String 타입을 반환하는 메소드에서 발생할 수 있는 캐스팅 에러(ClassCastException)를 방지하기 위해 컨버터 적용 대상에서 제외하는 설정이 필수적임.

## 멘토 피드백 및 시연 중심 설계
### 부스 시연 전략 및 우선순위 조정
부스가 위치할 실내 환경의 특성상 GPS 오차가 발생할 수 있으므로, 매끄러운 시연을 위한 가상 위치 이동 로직 구현이 선행되어야 함. 파손 및 분실 보상 기능은 후순위로 미루고 시연이 가능한 핵심 비즈니스 로직 위주로 개발 우선순위를 재설정함. 대면 픽업 및 수령 시 필요한 서명 기능은 클라이언트에서 SVG 형식으로 이미지를 전송하는 구조로 개발을 진행함.

### 배달 및 웹소켓 통신 로직
웹소켓은 배달 중 연결 끊김 시 복구 난이도가 매우 높으므로, 매칭 과정에서만 제한적으로 사용하기로 합의함. 배달 중 위치 갱신은 5초 주기의 API 폴링 방식으로 처리하고, 클라이언트 단에서 애니메이션을 통해 부드럽게 보정함. 배달 기사(드리미)가 앱을 종료하거나 통신이 끊길 경우에 대비해 재접속 대기, FCM 알림, 문자 발송 등의 복구 시나리오를 마련함. 추후 주문의 상태 전이(State Machine) 과정에서 필요한 입출력 데이터를 보다 명확히 정의할 계획임. 배달 비용은 우체국 모델과 유사하게 고정 비용을 기반으로 하되, 사용자가 상황에 따라 비용을 일부 조정할 수 있는 기능을 추가 검토함.
