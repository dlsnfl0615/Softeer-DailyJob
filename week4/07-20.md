## JPA vs JDBC
### JDBC Template
- 특징
  - SQL을 개발자가 직접 작성. 중복 코드(Connection 관리, PreparedStatement, ResultSet 파싱, 예외 처리 등)를 스프링이 대신 처리해 주는 최소한의 추상화 라이브러리
  - 통계, 대량 배치, 복잡한 다중 조인 및 벌크 수정/삭제가 주를 이루는 시스템에서 사용
  - 테이블 구조가 객체 모델과 1:1로 깔끔하게 떨어지지 않거나, 기존 DB legacy SQL을 그대로 가져와 써야 할 때 사용
- 장점
  - 복잡한 영속성 컨텍스트 개념이나 프레임워크 학습 곡선이 거의 없어서 가볍고 직관적임
  - 작성한 SQL이 그대로 실행되므로 튜닝이나 복잡한 쿼리 작성이 쉬우며 예측 가능함
- 단점
  - CRUD 작업도 매번 SQL을 직접 작성해야 함
  - 조회 결과를 객체로 매핑해주는 RowMapper 코드를 일일이 작성해 주어야 함


 ### Spring Data JDBC
- 특징
  - JPA의 복잡함을 배제한 DDD 기반 ORM
  - JPA의 복잡한 상태 관리(지연 로딩, 영속성 컨텍스트)가 부담스럽고, 가볍고 명확한 ORM을 선호할 때 사용
  - 도메인 간의 경계(Aggregate)가 명확하게 구분되는 구조일 때 사용
- 장점
  - JPA의 지연 로딩이나 상태 변화에 따른 사이드 이펙트가 없음. 엔티티 조회하면 연관된 aggregate 한번에 가져오므로 명확함
  - JPA 대비 학습 난이도 훨씬 낮고 빠른 개발 가능
  - Repository 인터페이스만 선언하면 기본 CRUD 메서드 자동 제공
- 단점
  - 1차 캐시나 변경 감지 같은 고성능 ORM 기능이 없음
  - 관련 정보가 많이 없음


 ### Spring Data JPA
- 특징
  - 자바 객체와 DB 테이블을 직접 매핑하며 SQL을 직접 작성하지 않고 객체 단위로 데이터를 조작함
  - 비즈니스 로직이 복잡하고, 객체 중심의 도메인 모델링이 중요한 일반적인 상황일 때 사용
- 장점
  - 기본적인 CRUD는 SQL 작성 없이 메서드 호출만으로 처리 가능
  - Dirty Checking 을 통해 엔티티 값만 바꿔도 자동 저장됨
  - 객체지향적 도메인 모델링 가능
  - 영속성 컨텍스트를 통한 1차 캐시, 쓰기 지연, 지연 로딩을 활용해 DB 접근 횟수를 줄일 수 있음
- 단점
  - 영속성 컨텍스트 생명주기, N+1 문제, 연관 관계 매핑, JPQL, Querydsl 등 제대로 다루기 위해 배워야 할 개념이 많음
  - 통계, 대량 배치, 복잡한 통계성 조인 쿼리를 순수 JPA로만 처리하기는 어려움

## 회고

### Keep (좋았던 점)
* **명확한 기술 근거 수립**: 결제-주문 구조, 전자서명 법적 효력 이슈, Soft Delete 정책 등 발생 가능한 예외 상황에 대해 논의하고 명확한 도메인 규칙을 정함.
* **상태 흐름 시각화**: State Machine 생성을 통해 단순 매칭 성공 케이스뿐만 아니라 분쟁, 사고(DISPUTED), 전문기사 인계(PARTNER_HANDOFF_PENDING) 등 복잡한 예외 상태를 사전에 도출함.

### Problem (아쉬웠던 점 및 해결 방안)
* **개발 스택 선정 시 제약 사항**: 모바일 웹 앱 기반(React Native/Capacitor 등) 구현 시 라이더의 실시간 위치(GPS) 추적에 기술적 한계가 있음을 인지함.
  * **해결 방안**: 현 단계에서는 스펙 범위를 명확히 조정하여 현실적인 MVP 구현에 집중하기로 합의.
* **데이터 접근 방식 검토 필요**: JPA와 JDBC 사용 사이에서 객체 단위 쿼리 vs 단일 속성 수정 간의 효율성 고민이 발생함.
  * **해결 방안**: JPA 구조 및 ORM 학습을 병행하여 내일 기술 스택을 최종 확정하기로 결정.
