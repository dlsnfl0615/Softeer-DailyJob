### NOT NULL과 default value
- NOT NULL + default

- NULLABLE + default
  - 기본값은 존재하지만 미정이나 없음의 의미로 저장해야 하는 경우 사용
 
### SSE vs WebSocket
| 구분 | SSE (SseEmitter) | WebSocket |
| --- | --- | --- |
| **통신 방향** | **단방향** (서버 $\rightarrow$ 클라이언트) | **양방향** (서버 $\leftrightarrow$ 클라이언트) |
| **프로토콜** | 표준 **HTTP / HTTPS** | **WebSocket 프로토콜** (`ws://`, `wss://`) |
| **연결 유지** | 단일 HTTP 연결 (HTTP Streaming) | HTTP Handshake 후 전용 프로토콜 전환 |
| **데이터 형식** | **텍스트 데이터만** (UTF-8, JSON 등) | **텍스트 및 바이너리 데이터** 모두 지원 |
| **재연결 지원** | 브라우저 기본 내장 (자동 재연결) | 직접 구현 필요 (JS 핸들링) |
| **방화벽 / 프록시** | 일반 HTTP 기반이므로 친화적 | 별도 포트/프로토콜 차단 이슈 가능성 존재 |

## 시스템 아키텍처 및 세션 관리

### 세션 기반 인증 및 중복 로그인 방지

분산 환경이 아닌 단일 서버 구조이므로 Spring의 `HttpSession` 및 톰캣 내장 세션 메커니즘을 활용하기로 결정함. 서버 재시작 시 세션 소실 문제를 보완하고 로그인한 유저를 빠르게 조회하기 위해 `{UserId : SessionId}` 맵을 별도로 유지하기로 함. 이를 통해 특정 유저가 다른 세션으로 중복 접근하는 문제를 방지하고, 세션 ID 기반 역방향 조회의 효율성을 높임. 한편, 장시간 소켓 연결 시 누적될 수 있는 `TIME_WAIT` 소켓 상태의 리소스 부하 문제에 대해서도 사전 고려함.

### 기본키 설계 및 정렬 가능 UUID v7 검토

로그인 아이디 대신 별도의 대리키(PK)를 두는 방향으로 의견을 모음. 브라우저 식별자 노출 방지, 아이디 변경 시 연쇄 수정(Cascade) 위험 차단, 개인정보 파기 및 가명화 용이성이 주요 근거임. PK 채택 시 인덱스 파편화 방지 및 타임스탬프 정렬이 가능한 **UUID v7** 도입을 검토 중이며, 현재 개발 환경(Spring, Java)에서의 호환 여부를 사전 확인하기로 함.

## DB 물리 설계 및 데이터 타입

### DDL 물리 설계 및 컬럼 규격 정의

팀 전체가 참여하여 ERD 물리 설계를 진행함. 주요 컬럼의 데이터 타입을 확정하였으며, 전화번호는 국가번호 및 하이픈을 포함해 `VARCHAR(50)`, 이메일은 표준 최대 길이를 고려해 `VARCHAR(255)`로 지정함. 평점 데이터의 경우 `TINYINT` 범위 초과 우려가 제기되었으나, Java 매핑 시 `int` 변환이 가능하고 SQL `SUM()` 집계 시 상위 타입으로 자동 승격되므로 가독성과 저장 공간 측면에서 적절함을 확인함.

### 위치 기반 서비스 및 상태 파악 데이터

위치 및 이동 시간 예측은 카카오 도보 길찾기 API를 활용하기로 결정함. 사용자 좌표 기반의 무분별한 API 호출 비용을 줄이기 위해 단순 시간 경과 기준 프로토타입을 먼저 구현한 뒤, 실제 거리 기반 로직으로 확장할 예정임. 거리 및 시간 예측 데이터는 '부르미-드리미 간 예상 소요 시간'과 '픽업 후 출발지-도착지 간 예상 소요 시간'으로 구분하여 DB에 분리 저장함.

## 비즈니스 로직 및 결제·인증 시나리오

### 결제 및 포인트 시스템 도입

카드 간편결제 시 매번 결제 정보를 입력해야 하는 UX 불편을 해소하고, 추가 요금 결제 흐름을 매끄럽게 처리하기 위해 **포인트 선충전 방식**을 도입함. 시연 편의성을 위해 신규 가입 시 3만 포인트를 기본 제공하며, 시연 과정에서 PG사(KG이니시스 등) 결제 및 자동 환불 흐름을 자연스럽게 보여줄 수 있도록 결제 시나리오를 구성함. 추가로 휴대폰 본인 인증 절차를 도입하기로 함.

### 서비스 자격 검증 및 보안 로직

드리미(배달기사)의 신원 검증을 위해 범죄경력회보서 제출 절차를 모의 구현함. 기관 코드 연동의 한계를 고려해 서류 사진 업로드 방식으로 대처함. 계정 로그인 시에는 서비스 정지 기간 컬럼을 조회하여, 정지 상태인 유저의 접근을 차단하는 보안 로직을 추가함.

## UI/UX 및 시연 최적화

### 사용자 중심 인터페이스 개선

배달 진행 시 수신자와의 실시간 소통이 핵심 요소라 판단하여, 화면 좌우 배치 시 연락 기능을 우선적으로 노출하도록 설계함. 배차 수락 시에는 연속 클릭 방지를 위한 쿨타임 UI를 적용함. 배달원 화면에는 픽업 및 수령 시 대면/비대면 여부와 필수 확인 정보를 채워 보여주는 반면, 주문자 화면은 정보 밀도를 줄여 간소화함.

### 접근성 및 시연 완성도

초기 단계부터 색상 대비 규격을 준수한 유니버설 디자인(Universal Design)을 적용하여 접근성을 강점으로 제시할 계획임. 시연 직전 마지막 단계에서는 실제 모바일 앱 형태의 웹뷰 패키징(Wrapper) 적용 여부를 검토하기로 함.
