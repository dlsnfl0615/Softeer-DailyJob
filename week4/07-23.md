

## 추가 학습
### RestClient
- SpringBoot 3.2 부터 도입
- 동기식
- get(), post(), put(), delete(): HTTP 요청 메소드 설정. post()나 put()의 경우 .body(...)를 통한 요청 본문(Payload) 전달 가능.
- uri(...): 요청 대상 URI(URL) 지정. 문자열 URL, URI 객체, 또는 템플릿 기반 Path Variable 바인딩 기능 지원.
- header(), contentType(), body(): HTTP 요청 헤더, 데이터 타입(Content-Type), 전송 객체 설정. retrieve() 호출 이전에 작성.
- retrieve(): 조립된 HTTP 요청을 서버로 전송하고 응답 수신 단계로 전환. 요청 스펙을 응답 스펙으로 변경하는 핵심 지점.
- body(Class<T>): 수신한 HTTP 응답 본문(JSON 등)을 지정한 자바 객체(DTO)로 변환(Deserialization)하여 반환.
  - 이때 사용하는 DTO 필드는 response로 들어오는 json의 구조와 깊이, 키 이름이 모두 동일해야 정상적으로 값을 받아올 수 있음.
- onStatus(...): HTTP 상태 코드(4xx, 5xx 등) 조건에 따른 커스텀 예외 처리 및 에러 로직 수행.
```java
// RestClient 객체 생성 (기본 빌더)
RestClient restClient = RestClient.builder().build();

// GET 요청 수행
KakaoWalkResponseDto response = restClient.get()
        .uri(targetUri)                                  // HTTP 요청 URI
        .header("Authorization", "KakaoAK " + apiKey)     // HTTP 헤더 추가
        .retrieve()                                      // 응답 수신
        .body(KakaoWalkResponseDto.class);              // 응답 바디를 자바 DTO 객체로 자바 바인딩
```

### UriComponentBuilder
- 안전하고 유연하게 HTTP URI 문자열을 조립해줌
- URL 쿼리 파라미터에 한글이나 특수문자, 공백이 포함되어 있으면 자동으로 UTF-8로 처리함
```java
URI uri = UriComponentsBuilder
        .fromUriString("https://dapi.kakao.com/v2/routing/walk") // 베이스 URL
        .queryParam("start_x", "127.1111")                     // ?start_x=127.1111
        .queryParam("start_y", "37.3947")                      // &start_y=37.3947
        .queryParam("end_x", "127.1262")                       // &end_x=127.1262
        .queryParam("end_y", "37.4199")                       // &end_y=37.4199
        .build()                                               // URI 빌드 (자동 인코딩)
        .toUri();                                              // URI 객체 반환

// 생성된 결과 URI: https://dapi.kakao.com/v2/routing/walk?start_x=127.1111&start_y=37.3947&end_x=127.1262&end_y=37.4199
```

### @RestController
- @Controller + @Responsebody 형태임
- @Controller는 반환값으로 html view 이름을 반환하는데 @RestController는 메소드가 리턴하는 객체를 http response body에 직접 json 형태로 변환해서 반환

### @RequestParm
- http request의 query parameter나 form 데이터를메소드의 매개변수로 바인딩
- 주로 URL 뒤에 붙는 ?key=value&key2=value2 형태의 데이터를 추출할 때 쓰임
- 속성
  - name / value: 요청 파라미터의 이름을 지정(메소드 변수명과 요청 파라미터명이 같으면 생략 가능)
  - required: 해당 파라미터가 필수인지 여부를 설정. 기본값은 true이며, 파라미터가 들어오지 않으면 400 Bad Request 에러가 발생
  - defaultValue: 파라미터 값이 전달되지 않았을 때 대신 들어갈 기본값을 문자열 형태로 지정. (defaultValue를 설정하면 required 속성은 자동으로 false처럼 동작함)
```java
@GetMapping("/search")
public String search(
        @RequestParam String keyword,
        @RequestParam(required = false) String category, // 필수 입력 아님 (안 들어오면 null)
        @RequestParam(defaultValue = "1") int page        // 안 들어오면 기본값 "1" 적용
) {
    return "Keyword: " + keyword + ", Page: " + page;
}
```

## 데이터베이스 설계 및 아키텍처 의사결정
### 원장(Ledger) 중심 데이터 모델링 및 SSOT 설정

* **원장 데이터의 원천성(SSOT) 확보**: `UserAccount` 테이블의 상태 컬럼을 단독으로 수정하는 방식 대신, `Ledger`(원장) 및 `Transaction`(거래) 테이블의 기록을 단일 진실 공급원(SSOT)으로 정의함
* **집계 잔액 갱신 방식**: `UserAccount`(지갑) 테이블은 조회 성능 향상을 위한 요약 용도로 사용하며, 거래 발생 시 원장 테이블의 누적 합산 결과를 반영하도록 구조화함
* **파생 컬럼(Derived Column) 제거**: 쿼리로 계산 가능한 파생 컬럼을 제거하고 잔액 스냅샷 및 변경 이력을 분리하여 회계적 정확성 확보함

### 지갑(Wallet) 및 자금 구조 이원화

* **Point와 Money의 역할 분리**:
* `Point`: 부르미 전용 결제 및 충전 수단
* `Money`: 드리미 전용 배송 수행 수익금 및 출금/환전 수단


* **단방향 환전 정책**: `Money`에서 `Point`로의 전환만 허용하며, 명시적 환전 요청 시 통합 트랜잭션 내에서 처리함

## 서비스 라이프사이클 및 자금 흐름
### 부르미 결제 및 환불 라이프사이클 (Point)

* **즉시 차감 및 거래 생성**: 퀵 서비스 등록 시 `Point`가 지갑에서 차감되며 거래 상태는 `PAID`로 기록됨
* **독립적 환불 거래 처리**: 환불 발생 시 기존 결제 데이터를 수정하지 않고 `REFUND` 형태의 새로운 장부 항목을 추가하는 불변(Append-Only) 패턴 적용함
* **주문 상태별 환불 정책 정의**:
* `MATCHING` ~ `WAITING_CONFIRMATION`: 정책에 따른 전액 또는 부분 환불 허용
* `WAITING_CONFIRMATION` 이후 (`CLAIM_REVIEW`, `SETTLEMENT_PENDING` 등): 일반 환불을 불가하며 클레임 조정 절차로 이관함

### 드리미 정산 및 수익 라이프사이클 (Money)

* **정산 대기(PENDING) 수익 생성**: 배송 완료 후 `WAITING_CONFIRMATION` 진입 시 `PENDING` 상태의 수익이 발생하나 가용 잔액에는 즉시 반영되지 않음
* **운영자 정산 프로세스**:
* **승인**: `SETTLEMENT_COMPLETED` 거래 기록 및 `PENDING` $\rightarrow$ `AVAILABLE(SETTLED)` 상태 전환을 통해 가용 잔액으로 이동함
* **거절/클레임**: `SETTLEMENT_REJECTED`, `REVERSAL`, `CLAIM_ADJUSTMENT` 거래를 생성하여 정산 취소 및 금액 조정 내역을 보존함


### ORM 및 데이터 접근 기술 의사결정

* **JPA 우선 채택**: 제한된 프로젝트 기간 내 생산성을 최우선으로 고려하여 JPA를 기본 persistence 프레임워크로 선정함 (#8 이슈)
* **JDBC Template 병행 검토**: 향후 부하 테스트(Stress Test) 진행 후 잔액 집계 및 대량 트랜잭션 병목 구간 발생 시 JDBC를 선택적으로 병행하기로 합의함
* **UI/UX 세부 요소 반영**: 은행 잔고 표시 영역에 신뢰성 전달을 위한 인증 표시 요소 반영 결정함
