### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 로그인 여부 검증을 거치는 /mypage 요청과 달리, 사용자가 주소창에 /mypage/index.html 또는 /registration/index.html 등 정적 파일 경로를 직접 입력하면 로그인 없이도 접속이 허용되는 정적 리소스 우회 접근 문제 발생.
- 처음 추측한 원인: HttpActions에 별칭(alias)을 매번 추가하는 방식은 맵에 등록할 경로가 너무 많아 비효율적이라 판단, URL 문자열에 .html 포함 여부나 끝자리 확장자를 검사하여 일괄 차단(메인 페이지 제외)하려고 시도함. 그러나 302 redirect 정상 요청까지 차단되거나 /mypage/처럼 슬래시(/)로 끝나는 폴더 경로 우회를 막지 못하는 한계 직면.
- 원인을 확인하기 위해 시도한 방법: AuthRedirectHandler를 삭제하고 StaticFileHandler 내부에서 로그인 체크가 필요한 폴더 경로를 리스트로 선언하여 관리하도록 구조 변경. 이후 /registration/index.html 경로로도 동일한 우회 접근이 발생하는지 재차 테스트를 수행함.
- 실제 원인: 인증 핸들러와 정적 파일 핸들러의 역할이 명확히 분리되지 않아 발생한 문제임. 보안이 필요한 특정 정적 리소스(HTML) 경로로 요청이 직접 들어올 때, 로그인 상태별로 접근 권한을 제어하는 분기 로직이 정적 파일 처리 엔진(StaticFileHandler) 단에 부재했기 때문임.
- 해결 방법: StaticFileHandler 내부에 사용자의 로그인 상태(세션 존재 여부)에 따른 접근 권한 분기 로직을 직접 추가함. 이를 통해 권한이 없는 비인증 사용자가 정적 HTML 파일 및 하위 폴더 경로를 찌를 때의 우회 통로를 완벽히 차단함.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 특정 리소스 및 폴더에 대한 접근 제어(Access Control) 규칙이 정적 파일 핸들러 영역까지 공통 미들웨어 형태로 안전하게 적용되어 있는지, 권한별 진입점 분기 로직을 먼저 검증할 것.
