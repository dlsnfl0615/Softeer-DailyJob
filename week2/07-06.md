### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: OSI 7계층과 TCP/IP 4계층 학습. DNS, JWT 학습. Http response 상태 코드 반환.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 쿼리 파라미터 디코딩 구멍. 현재는 퍼센트 기호 + 숫자 2자리 형태만 디코딩할 수 있음

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: `HttpStatus` enum 선언한 것
- 그렇게 판단한 이유: step 4 요구사항을 보니 앞으로도 상태 코드 반환하라고 시킬 것 같아서 아예 클래스로 따로 빼두고 응답 메시지 작성하면 더 편할 것 같았음
- 고려했지만 선택하지 않은 대안: 기존처럼 메소드를 상태코드에 대해서 전부 다 만들기
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: `HttpStatus`에 상태코드만 추가하면 메소드가 하나씩 추가되는 것과 같은 현상. 기능 추가하기도 좋다

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: HTTP Request Message를 파싱할 때 Body의 데이터가 깨지거나, 서버가 응답을 보내지 못하고 무한 대기 상태에 빠지는 현상 발생
- 처음 추측한 원인: `BufferedReader` 하나만으로 Header와 Body를 모두 처리할 수 있을 것으로 추정하고 `readLine()`을 활용해 파싱을 시도함
- 원인을 확인하기 위해 시도한 방법: Header는 줄바꿈 단위로 끊어지므로 `BufferedReader`로 읽고, Body는 `InputStream`을 이용해 Content-Length 바이트 수만큼 직접 읽도록 구조를 변경함
- 실제 원인: `BufferedReader`가 내부 버퍼링(기본 8KB)을 수행하면서 Header를 읽을 때 소켓 스트림에 남아있던 Body의 앞부분 데이터까지 미리 당겨서 읽기 때문에 body의 내용을 온전히 읽어올 수 없었음
- 해결 방법:
  - 채주혁님의 PR(https://github.com/softeerbootcamp-8th/be-was/pull/90) 소스코드를 참고하여, 텍스트 기반의 `BufferedReader`를 완전히 배제하는 힌트를 얻음
  - 최종적으로 Header와 Body 영역 모두를 `InputStream`을 이용해 바이트 단위로만 순수하게 읽어오도록 통일하여 파싱에 성공함
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 네트워크 소켓 스트림을 제어할 때 하나의 스트림에 대고 성격이 다른 보조 스트림(`BufferedReader`)과 기반 스트림(`InputStream`)을 혼용하여 버퍼 침범 문제가 발생하지 않았는지 먼저 체크 예정

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 두 번의 설계 실패로 구현이 막혔을 때, 바로 AI를 통해 정답을 찾지 않고 동료들의 PR 코드를 적극적으로 찾아보며 `BufferedReader` 내부 버퍼 동작의 구조적 한계를 빠르게 파악하고 해결책을 찾아냄
- 그 접근이 도움이 된 이유: `BufferedReader`가 내부적으로 어떻게 동작하는지 알게 됨. AI에게 딸깍으로 물어보는 것도 좋지만 다른 사람들이 어떻게 작성하는지 코드 흐름을 읽어가면서 원인과 해결방법을 파악하는 것도 도움이 된다는 걸 알게 됨
- 다음에도 반복하고 싶은 점: 공식 문서나 다른 사람의 코드로부터 해결방법 찾아보기
