### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: /registration 요청 시 GET is not accepted for registration 오류가 발생했으며, 이를 우회하려고 루트 페이지의 버튼을 숨겨진 POST 폼으로 바꾸자 이번에는 데이터 필드 누락으로 인한 NPE (null userId) 가 발생함.
- 처음 추측한 원인: 프론트엔드 버튼의 HTML 마크업 태그나 전송 방식에 단편적인 문제가 있을 것으로 추측함.
- 원인을 확인하기 위해 시도한 방법: 디버깅 및 라우팅 키 매칭 과정을 역추적함.
- 실제 원인: URILinker로 변환된 최종 파일 경로(/registration/index.html)를 라우팅 키로 혼용하다 보니, "페이지를 단순히 보여주는 GET 요청"과 "회원가입 데이터를 처리하는 POST 액션"이 동일한 키를 두고 충돌을 일으킨 것이 근본 원인이었음.
- 해결 방법: 정적 페이지 요청은 StaticFiles 엔드포인트로 두고, 데이터 처리 액션은 /api/register와 같이 별도의 명시적인 네임스페이스(ApiActions) 경로로 완전히 분리함.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 화면 진입(GET)과 비즈니스 로직 처리(POST)가 혹시 동일한 경로 키로 결합되어 라우팅 테이블에서 충돌하고 있지 않은지 최우선으로 확인할 것.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 테스트 코드(RoutingHandlerTest.StaticFileHandler_test)를 수리하는 과정에서 StaticFileHandler가 실제 포맷팅된 경로(.html)와 변환 전 bare 경로 리스트를 비교하여 인증 게이트가 완전히 죽어있던 논리 버그를 포착하고 수정한 점.
- 그 접근이 도움이 된 이유: 기능 구현에만 매몰되지 않고 컴파일이 깨져있던 스텁 테스트를 완성하려 시도했기 때문에, 눈으로 확인하기 힘든 런타임 논리적 오류를 사전에 잡아낼 수 있었음.
- 다음에도 반복하고 싶은 점: 코드가 돌아간다고 방치하지 말고, 작성된 테스트 코드를 완벽하게 구동 가능한 상태로 유지하며 구현부를 교차 검증하는 버릇을 들일 것.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: new TreeMap<>(String.CASE_INSENSITIVE_ORDER)를 적용해 두고 내부의 .get("Cookie"), .get("Content-Type") 코드를 대소문자가 혼용된 하드코딩 형태로 내버려 둔 점.
- 그렇게 된 원인: 재 맵 구조가 대소문자를 무시하도록 구현되어 있어 당장 에러가 나지 않다 보니 심각성을 인지하지 못함.
- 다음에는 다르게 해볼 구체적인 행동: TreeMap을 사용하는 가장 큰 목적은 정렬임. 단순히 대소문자만을 위해 사용하면 오버엔지니어링임. 클래스의 사용 목적을 명확히 알고 사용해야 함.
