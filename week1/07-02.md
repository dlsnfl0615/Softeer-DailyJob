### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: `QueryParser`에서 퍼센트 인코딩된 값 정규식으로 찾은 것
- 그렇게 판단한 이유: Pattern과 Macther를 사용하니까 오류 생기면 잡기 수월함
- 고려했지만 선택하지 않은 대안:
  - 퍼센트 기호 위치 찾아서 2칸 뒤까지 포함해 총 3칸을 원래의 기호로 대체하는 방법
  - `loc = str.indexOf("%")` 이런 형태가 될텐데 나머지 코드들까지 포함하면 너무 길어짐
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 한눈에 알아보기 쉽고 pattern 변수 수정하면 정규식 수정 가능

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: html get 요청 보내는거
- 처음 추측한 원인: html form, input, button 등 너무 오래전에 해서 기억 안남. 다시 정보 찾아봄
- 원인을 확인하기 위해 시도한 방법: 각 input 태그의 값들을 자바스크립트에서 합친 다음에 제출 버튼에서 이 값을 보내도록
- 실제 원인: html 태그에 속성 알맞게 설정해주면 알아서 쿼리 만들어줌 
- 해결 방법: input 태그의 부모 form에 action을 create로, method를 get으로 설정. 각 input 태그에 name을 주어진 요구사항에 맞게 작성하면 알아서 쿼리 만들어줌
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: html 복습하자

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: AI에게 물어보기 전에 사람들이 작성한 블로그와 자바 공식 문서 확인한 후에 도저히 답이 없을때 물어본 것
- 그 접근이 도움이 된 이유: 처음부터 AI에게 물어보면 요구사항에 오버되는 해결 방법 알려주거나 내가 개발하려는 방향과 다른 방식으로 알려주는 경우가 많은데, 큰 골격을 머리에 넣어놓고 작은 단위만 AI에게 질문하니까 구조를 크게 해치지 않는 선에서 궁금한 점을 물어볼 수 있었음
- 다음에도 반복하고 싶은 점: 앞으로 클로드 코드 키 받을텐데 답지 확인하는 용도로만 사용하고 최대한 내가 먼저 해결해보자

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 테스트코드가 순수 자바 코드만을 이용해서 작성해야 하는건데 이렇게 웹 띄워서 작동하는 프로젝트의 경우에는 테스트코드를 어떻게 작성해야 하는지 모르겠음
- 그렇게 된 원인: 핵심 기능 구현만을 위해 개발하는 것도 바빠 죽겠는데 테스트코드 작성법까지 추가로 학습하려고 하니까 시간이 없음
- 다음에는 다르게 해볼 구체적인 행동: 주말에 테스트코드 클래스, 메소드 등 공부하자

# 추가 학습
> 아래는 금일 수업 내용 필기 및 과제 진행 과정에서 알게 된 내용 정리한 것.

### 모델링

- 모델링 먼저 하기
    - 애매한 부분 먼저 발견
    - 기획 개발 DBA 등 관계자가 같은 그림 보고 대화
    - 잘못된 가정에서 오는 위험과 재작업 감소
    - 빠른 의사결정, 빠른 가시화
- 모델링
    - entity: thing
    - attribute: 성격 정보
    - relationship: 사이의 연결
- entity
    - 관리해야 하는 대상. 명사
    - 인스턴스의 모임
    - 엔티티 = 표, 인스턴스 = 행
    - 따로 저장하고 추적할 필요가 있는가. 업무에 필요한가. 여러개가 모이는가. 식별이 되는가
- relationship
    - 회원 —<주문>— 상품
    - 1 N M
- M:N은 "언제" 푸는 시점
    - 관계 자체에 **속성** 을 채워야 할 때 (수량, 가격, 신청일...)
    - 관계가 또 다른 **자식 엔티티** 를 가져야 할 때
    - 관계명을 못 붙이겠다면, 그건 **이름 없는 엔티티** 가 숨어 있다는 뜻

### Entity

- 관리해야 하는 대상. 명사
- 인스턴스의 모임
- 엔티티 = 표, 인스턴스 = 행
- 따로 저장하고 추적할 필요가 있는가. 업무에 필요한가. 여러개가 모이는가. 식별이 되는가
- 식별 관계
    - 자식의 식별자(PK)의 일부가 됨
    - unique ↔ unique X
    
    - 이름을 키로 설정해버리면 가족 이름 겹치면 입사 불가
- 비식별 관계
    - 자식의 일반속성(FK) 이 됨
    - unique ↔ unique

- 회원이 상품을 사면 주문(행위)이 생긴다

### Super type/Sub type

회원은 반드시 하나의 서브 타입에 속해야 함

### Attribute

- 좋은 속성 기준
    - 원자값: 일단 최소로 쪼개고 필요에 따라 합치기
        - 업무의 요구사항에 맞게
    - 단일값: 회원이 전화번호 여러개면 `회원연락처`라는 자식 엔티티로 빼냄

### 기존 방식

- `Thread thread = new Thread(new RequestHandler(connection));` 에서 새로운 thread 요청 들어올 때마다 생성하니까 out of memory 발생 가능

### Executor

- 사용 이점
    - 사용할 수 있는 thread 최대 개수를 미리 설정해둬서 out of memory 방지
    - thread 재사용하니까 cpu 수준에서 삭제 생성 과정 없앰 → 효율적
    - context switching 감소

### Get

‘&’로 이어서 작성
