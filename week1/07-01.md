# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 기능요구사항 1과 2
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: concurrent로 변경

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 이미지와 css 파일 출력하기 위해서 따로 enum 정의한 것.
- 그렇게 판단한 이유: "요청하는 파일의 확장자" + "타입" 형태가 고정되기 때문에 클래스로 만들어서 재사용하는 게 나아보임. 그리고 오늘 강의시간에 언급된 enum 실제로 실습해볼 수 있는 기회.
- 고려했지만 선택하지 않은 대안: 이거 말고 떠올린 방법은 if else문 덕지덕지 붙이는 방법
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 다른 파일 형식이 들어오면 컨트롤러쪽 코드 수정할 필요없이 ContentType만 수정해서 기존 코드 변경하지 않고 사용 가능.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 이미지 아이콘이 출력 안 됨
- 처음 추측한 원인: resource/static 경로를 못찾나?
- 원인을 확인하기 위해 시도한 방법:
  - `byte[] body = Files.readAllBytes(new File("src/main/resources/static" + url).toPath());` 로 작성한 곳에서 `src/main/resources/static` 이렇게 경로를 지정한게 절대 경로로 들어가는거라고 생각
  - `src/main/resources/static` 을 `./static` 으로 변경
  - `23:41:13.878 [ERROR] [Thread-0] [webserver.RequestHandler] - ./static/index.html` 뜨면서 화면에 "페이지가 작동하지 않습니다." 출력됨.
- 실제 원인: `ContentType` 클래스에서 from 메소드 정의할 때 name이랑 type을 반대로 작성해서 제대로된 문자열을 반환받지 못하던 것.
- 해결 방법: name <-> type 위치 변경
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 전체 실행 자체에는 문제가 없고 디버깅 코드에서도 문제가 발견되지 않았음. 그럼 컨트롤러 쪽에는 문제가 없다는 것이니까 domain 쪽을 살펴봐야 함.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: if else문 남발해서 쓰지 않고 클래스 정의한 것
- 그 접근이 도움이 된 이유: 가독성도 좋아지고 유지 보수도 좋아짐
- 다음에도 반복하고 싶은 점: 가장 쉬운 해결책보다 미래를 위해 유지보수에 유리한 점이 있을지 생각해보고 개발하는 것.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: `RequestHandler`에서 run 메소드의 파싱 부분에서 처음에 hostValue와 cookieValue를 null로 설정하고 반복문 안에서 값 덮어쓰게 하고 있음.
- 그렇게 된 원인:
  - 어쨋든 반복문 안에서 한 줄씩 읽어야 하는거고 `Header`에 생성자로 넣으려면 바깥에 변수가 선언되어야 하는건 맞으니까 이렇게 했음.
  - 이렇게 하다보니까 String에 null이 들어가 있고 이게 반복문 안에서 덮어써지지 않으면 null 그대로 들어가게 됨.
  - null일 때 처리하는 로직을 추가하거나 구조를 더 좋게 바꾸는게 좋아보임.
- 다음에는 다르게 해볼 구체적인 행동: try catch나 Optional처럼 예외처리에 대해서 더 신경써야 함.
