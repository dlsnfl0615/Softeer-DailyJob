# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: "/../../../" 등의 주소를 요청해서 허용해준 경로가 아닌 상위 경로에 접근하는 것 막기
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 테스트 코드 작성, `HttpStream`, `HttpRequest`, `HttpResponse` 구조

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: `getCanonicalFile` 을 먼저 적용한 후에 static 경로와 합치기
- 그렇게 판단한 이유: 기준 경로가 없는 상태에서 계산하기 때문에 앞의 "/../../../"이 의미가 없어짐
- 고려했지만 선택하지 않은 대안: static 경로와 url을 합친 후에 마지막에 `getCanonicalFile` 을 하면 먼저 합치는 과정에서 상위 디렉토리로 이동해버림
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 공격에 안전하게 디렉토리 이동할 수 있게 됨

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: RequestHandler에 너무 많은 책임이 들어가 있음
- 처음 추측한 원인: 기능 구현을 우선시 해서 일단 RequestHandler에 다 때려박음
- 원인을 확인하기 위해 시도한 방법: 책임 분할
- 실제 원인: 처음 추측과 동일
- 해결 방법:책임 분할
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 기능 구현 먼저 하고 리팩토링하면서 책임 분할하는게 더 비효율적인 것 같아서 기본 골자는 어느정도 갖춰졌으니 미리미리 객체 지향적 설계를 하자

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 다른 사람들 PR 훔쳐보기
- 그 접근이 도움이 된 이유: 테스트 코드 어떻게 작성하는지, 커밋 메시지 어떻게 작성하는지, 구조 어떻게 짜는지 확인해볼 수 있었음
- 다음에도 반복하고 싶은 점: 가장 빠르게 PR올리는 사람이 보통 잘 하는 사람일테니까 많이 참고해보자

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 코드 작성부터 무작정 시작했던 점
- 그렇게 된 원인: 기능 구현만 생각하다보니 마음이 급해져서
- 다음에는 다르게 해볼 구체적인 행동: 일단 전체 구조를 다시 떠올려보고 아래로 내려가면서 필요한 책임을 분할해가며 클래스들에게 할당하는 방식으로 코딩해보기
