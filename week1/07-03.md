### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: `getCanonicalFile` 을 먼저 적용한 후에 static 경로와 합치기
- 그렇게 판단한 이유: 기준 경로가 없는 상태에서 계산하기 때문에 앞의 "/../../../"이 의미가 없어짐
- 고려했지만 선택하지 않은 대안: static 경로와 url을 합친 후에 마지막에 `getCanonicalFile` 을 하면 먼저 합치는 과정에서 상위 디렉토리로 이동해버림
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 공격에 안전하게 디렉토리 이동할 수 있게 됨

# 주간 프롬프트 회고 템플릿

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:
```
domain과 component 패키지를 만들었다고 했을 때 각 패키지에는 어떤 클래스들이 들어가는게 맞는지. 그리고 domain, component, model 이렇게 나누기도 하는지. model은 mvc에서 쓰는거니까 이렇게 3개로 나누는건 어색하지 않나
```

선택한 이유: 이번 주차에서는 간단하게 내가 모르는 내용 질문 용도로만 사용했음. 해당 프롬프트는 프로젝트 구조를 객체 지향적으로 바꾸는 과정의 시작점이 되는 부분이기 때문에

### 2. 원본 프롬프트 구조 분석
- 문제 상황: 백엔드 프로젝트 아키텍처 설계 중 domain, component, model 패키지의 역할 분담과 경계가 모호하여 패키지 구조의 적절성을 고민하고 있음.
- 원하는 결과를 어떻게 요구했는가: 각 패키지에 어떤 클래스가 들어가는지, 이 세 가지를 함께 나누는 설계가 어색한지(특히 MVC 관점에서 model의 중복성)에 대한 답변을 요구함.
- 필요한 맥락이 포함되어 있는가: 사용하고자 하는 기술 스택(Spring Boot 등)이나 아키텍처 스타일(DDD, Layered 등)에 대한 구체적인 맥락이 생략되어 있음.
- 판단 기준이나 제약 조건이 있는가: MVC 패턴의 관점에서 model과 domain이 겹치지 않는가에 대한 의구심이 판단 기준으로 작용하고 있음.
- 출력 방식이 명확한가: 질문 형태로 나열되어 있어, 구체적인 예시나 비교 테이블 등 원하는 답변 형식이 명시되지 않음.
- 나의 가설, 우려, 의도가 드러나는가: 'MVC의 model이 있는데 domain과 component를 함께 나누는 것이 어색할 수 있다'는 우려와 의문이 명확히 드러남.
