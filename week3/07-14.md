# be-DailyJob

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습:
  - `byte[]` 기반 바이트 보존 Multipart/form-data 수동 파서 구현 및 이미지 파일 업로드 기능 완성
  - H2 데이터베이스 내 `BLOB` 타입을 활용한 파일 바이너리 데이터 저장 구조 구축 및 연동
  - 댓글 작성 시 `feedId` 쿼리 파라미터 전달 및 원래 보던 게시글 위치 유지 기능 구현


- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 댓글에도 프로필 이미지 적용하기, 프로필 이미지 변경 기능 구현하기

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단:
  - `HttpRequest.readBody`가 요청 바디를 바로 UTF-8 String으로 변환하지 않고, raw `byte[]` 형태로 보존하게 한 뒤 기존 텍스트 기반 핸들러를 위해 `httpBody()` 편의 메서드를 만들어 점진적 리팩터링을 유도함.


- 그렇게 판단한 이유:
  - 바이너리 데이터(이미지 파일)를 문자열로 먼저 변환하면 원본 데이터가 손상되어 정상적인 파일 복원이 불가능함. 헤더를 포함한 전체 영역을 `byte[]` 파싱으로 전환하는 과도한 설계를 피하면서 필요한 바디 부분만 안전하게 바이트 수준으로 처리하기 위함임.


- 고려했지만 선택하지 않은 대안:
  - 요청 헤더 영역까지 전부 바이트 파싱 구조로 전면 전환하는 대안. 그러나 이는 구현 복잡도가 너무 크고 기존 코드의 안정성을 해칠 우려가 있어 제외함.


- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 바이너리 깨짐 없이 안전한 multipart 파일 처리가 가능해졌으며, 기존 텍스트 데이터 파싱 비즈니스 로직도 부서지지 않고 안전하게 하위 호환성을 유지하게 됨.



### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상:
  - 직접 구현한 `MultipartParser` 테스트 및 실 구동 과정에서 바이트 인덱스 범위 초과, 문자열 따옴표 섞임, 개행 문자 잔여, 구분자 미매칭 등 6가지 이상의 다양한 파싱 버그가 다발적으로 일어남.
  - JSON 렌더링 시 null 값 처리 거부나 이미 변환한 UTF-8 base64 바이트를 다시 변환하면서 이중 인코딩되는 현상이 발생함.


- 처음 추측한 원인:
  - 단순 오프셋 계산 실수이거나 외부 라이브러리(`/api/feedList` 등)의 일시적인 연동 문제로 추측함.


- 원인을 확인하기 위해 시도한 방법:
  - `Arrays.toString()`을 이용하여 가상 바이너리 바이트 배열의 수치 값을 로깅하고, 슬라이싱 구간의 오프셋을 하나씩 대조하는 디버깅용 유닛 테스트 코드를 작성해 디버깅함.


- 실제 원인:
  - 첫 번째 경계(delimiter)의 인덱스 0 조건 처리 누락, 헤더에 포함된 세미콜론과 등호 파싱 정규식의 경계 처리 미흡, 스트레이 `\r` 오프셋 버그 등이 원인이었음. JSON 렌더링 에러는 과거 DB 테이블에 저장되어 있던 content null 값과 중복 base64 직렬화 분기 처리 미흡으로 발생함.


- 해결 방법:
  - 0번 인덱스 바운더리 체크 예외를 제거하고 정규식 그룹 추출 방식을 정교화하였음. `Json.render` 내부에 null 검사 분기 로직을 삽입하고, 데이터 전달 단에서 `new String(file, UTF_8)`을 사용해 이중 인코딩 문제를 수정함.


- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - `byte[]`와 `String` 간 경계선에서 인코딩/디코딩이 언제 이루어지는지 체크하고, 디버깅 시에는 무조건 주소값만 찍히는 `toString()` 대신 `Arrays.toString()`이나 바이트 길이를 직접 찍어볼 것.



### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지:
  - 바디 파싱 로직 테스트를 작성할 때 장황한 수동 페이로드 배열 조립 대신 가변 인자를 받는 `concat(byte[]... arrays)` 유틸리티 헬퍼 메서드를 도입하여 작성함.


- 그 접근이 도움이 된 이유:
  - 가독성이 극적으로 개선되어 페이로드 구조를 한눈에 볼 수 있었고, 테스트 케이스 추가 시 중복 코드를 방지할 수 있었음.


- 다음에도 반복하고 싶은 점:
  - 지저분한 바이트 조작 테스트 코드를 짤 때는 미리 깔끔한 헬퍼 메서드 설계를 선행하여 테스트 본질에만 집중하기.



### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 예외 계층 구조를 섬세하게 설계하지 못해 `EditProfileHandler`에서 예외 발생 시 전용 예외 대신 기존 회원가입 전용 매핑 예외를 던져 405 Method Not Allowed 등의 엉뚱한 응답 에러를 보며 디버깅 시간을 허비함.
  - 화면 이동 스크립트 작성 시 인코딩을 명시하지 않아 깨지는 구조를 방치한 것.


- 그렇게 된 원인:
  - 예외 처리 및 공통 핸들러 인터페이스의 매핑 범위를 깊게 인지하지 못하고 손쉬운 기본 예외를 재사용하려고 함. 응답 바이트 전송 시 플랫폼 기본 인코딩 값에 부주의하게 의존함.


- 다음에는 다르게 해볼 구체적인 행동:
  - 오류 케이스를 분기할 때는 전용 비즈니스 예외 클래스(`ProfileEditFailedException` 등)를 확실히 정의하여 전용 상태 코드로 흐르게 만들 것. 브라우저 응답을 보낼 때는 언제나 `Content-Type: text/html;charset=utf-8` 헤더와 `StandardCharsets.UTF_8` 명시를 습관화할 것.
