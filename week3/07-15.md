# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:

  - 세션 데이터베이스(H2SessionDatabase)에 loginTime 컬럼을 추가하고 세션 생성 시간 저장 로직 구현. getSession 호출 시 현재 시간과 비교하여 설정된 만료 시간(expireMillis) 초과 시 세션을 자동 만료 처리하는 로직 구축 완료.

  - 로그인 상태와 로그아웃 상태에 따른 정적 리소스 및 API 요청의 접근 제한 분류 및 Enum 설계 리팩토링. StaticFiles Enum에서 로그인/로그아웃 필수 요청의 API 및 Path 경로를 모두 추출하는 flatMap 기반 Stream 메소드 완성.

  - 로그인 상태에서 로그아웃 필수 요청(/api/register 등)을 직접 GET URI로 우회 접근하거나 개발자 도구의 fetch 콘솔 요청을 통해 악의적으로 POST 요청을 보낼 때의 차단 로직(Spring Security 유사 필터링 구조) 검증 완료.

  - /logout 경로 접근 시 서버 자체 예외 메시지가 그대로 클라이언트에 노출되던 문제를 분석하고 RequestHandler 단의 NoSuchFileException 예외 처리(try-catch 후 커스텀 메시지 throw) 적용 완료.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분:

  - 로그인/로그아웃 여부와 상관없이 모두 접근 가능한 공통 API인 LIKE, NICKNAME, FEED_LIST, COMMENT_LIST 요청에 대한 권한 매핑(상태 바이트 2) 필터 단 처리 완료하기.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:

  - StaticFiles Enum의 getLoginRequirement()와 getLogoutRequirement() 메소드가 API 경로와 실제 파일 경로(Path)를 모두 아우르는 하나의 단일 리스트를 반환하도록 설계 변경. flatMap과 distinct를 결합하여 두 경로 유형을 동시에 관리함.

- 그렇게 판단한 이유:

  - 사용자가 주소창에 /home_main으로 요청하든, 실제 물리 경로인 /main/index.html로 우회 요청하든 두 경로 모두 동일한 로그인 여부 검증 대상이어야 함. 각각 리스트를 만드는 것보다 한 번에 평탄화(flatMap)하여 필터 검증 단에서 단순 contains 연산으로 확인하는 것이 안전하고 구조가 명확하기 때문임.

- 고려했지만 선택하지 않은 대안:

  - 단순 api 리스트만 가공하여 비교하고 path 요청은 필터에서 예외적으로 누락시키는 방식. 이 경우 사용자가 실제 파일의 물리 URI(index.html 등)를 직접 주소창에 치고 들어갈 때 보안 우회가 일어날 우려가 있어 제외함.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:

  - 필터 단의 인증 여부 비교 검증 로직이 극도로 단순화되어 한 줄의 contains만으로 처리할 수 있게 됨. 관리 포인트가 Enum 내부로 캡슐화되어 유지보수성이 증가함.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:

  - 로그인된 상태에서 정적 페이지에 접근할 때 의도하지 않은 서버 에러 메시지가 웹 브라우저 alert에 노출됨. 또한, 존재하지 않는 /logout 경로 요청 시 작성하지 않은 알 수 없는 예외 메시지가 클라이언트로 전달됨.

- 처음 추측한 원인:

  - StaticFiles Enum 매핑 기능인 getLoginRequirement에서 getApi 대신 getPath를 호출하고 있어서 파일 경로가 그대로 노출되어 에러가 난 것이라 추측함.

- 원인을 확인하기 위해 시도한 방법:

  - StaticFiles::getPath 매핑 코드를 getApi로 수정해 보았으나 여전히 파일 경로 관련 에러가 드러남. 호출 체인 구조를 추적하여 실제 물리 파일이 없는 경로를 호출했을 때의 예외 전파 과정을 추적함.

- 실제 원인:

  - /logout처럼 서버에 존재하지 않는 파일 경로를 요청받았을 때, Files.readAllBytes가 내부적으로 NoSuchFileException을 던짐. RequestHandler에서 이 JVM 레벨의 예외를 명시적으로 잡아서 핸들링하는 별도의 예외 메시지 처리가 없었기 때문에 날것의 예외 내용이 클라이언트로 그대로 전달된 것임.

- 해결 방법:

  - RequestHandler에서 NoSuchFileException 등이 발생할 수 있는 파일 입출력 로직을 try-catch로 감쌈. 예외 캐치 후 의미 있는 에러 메시지를 담은 예외를 새로 정의하여 명시적으로 throw 함으로써 사용자 정의 에러 메시지가 클라이언트에 반환되도록 구현함.

- 다음에 비슷한 문제가 생기면 먼저 확인할 것:

  - 클라이언트에 노출되는 에러 메시지가 있는 경우, 예외가 던져지는 가장 말단 지점(I/O를 처리하는 Low-level 라이브러리)의 예외가 위쪽 상위 레이어에서 적절히 정제 및 가공 처리가 되고 있는지 먼저 디버깅할 것임.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:

  - 단순히 브라우저 상의 GET 요청 차단만으로 "회원가입 불가"를 끝내지 않고, 개발자 도구의 Console 창에서 fetch로 POST 요청을 강제 전송하는 침투 테스트를 직접 수행해 봄.

- 그 접근이 도움이 된 이유:

  - 브라우저 제약사항(주소창 엔터 = GET)에만 의존하는 개발은 보안 취약점을 유발한다는 사실을 직접 증명할 수 있었고, 서버 도메인 로직과 필터에서 세션 상태(로그인 유무) 및 HTTP 메소드를 복합적으로 반드시 검증해야 한다는 명확한 근거를 얻음.

- 다음에도 반복하고 싶은 점:

  - 프론트엔드의 화면 단 흐름에 안주하지 않고, 임의의 API 요청 도구(개발자 도구 fetch, Postman 등)를 사용해 악의적 요청을 의도적으로 보내보는 API 취약점 검증 방식을 앞으로도 계속 생활화할 것임.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:

  - 개발자 도구의 콘솔에서 fetch로 요청을 테스트할 때, 서버 파싱 에러(배열 인덱스 초과 에러, ArrayIndexOutOfBoundsException)의 원인을 바로 찾지 못하고 단순 네트워크 요청 문제로 접근하여 시간을 허비함.

- 그렇게 된 원인:

  - 서버의 파서가 양식 데이터(application/x-www-form-urlencoded)를 파싱하기 위해 문자열을 = 혹은 & 기준으로 쪼개는 구조를 사용하고 있었는데, 전송 헤더를 application/json 포맷으로 보낼 때의 파싱 호환 문제를 조기에 자각하지 못함.

- 다음에는 다르게 해볼 구체적인 행동:

  - 서버에서 배열 범위 초과 에러가 나타나면, 전송한 요청 데이터의 직렬화 포맷(JSON / URL Encoded)과 서버가 수신하는 본문 바인딩 로직이 일치하는지 규격(Header) 정보부터 우선 대조하여 트러블슈팅 시간을 단축할 것임.
